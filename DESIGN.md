# RCサーキット自動順位検出 設計たたき台

## 目的

朽ち果てた植物園をオフロードRCカーのサーキットとして使い、ゴールライン通過順を自動記録する。Raspberry Pi + カメラでスタート/ゴール通過を検出し、順位、仮ID、通過時刻、車両画像を画面に表示する。

最初の実装では「車両が何台目にゴールしたか」を確実に取り、車両の恒久的な個体識別は補助機能として扱う。

## 結論

推奨構成は、横から固定したカメラ映像に対して、OpenCVの高速な動体検出をトリガーにして通過候補フレームを一時保存し、YOLOや画像処理はバックグラウンドで確認・切り抜きに使うハイブリッド方式。

順位はYOLOの処理完了順ではなく、カメラから取得したフレーム時刻とゴールライン横断時刻で確定する。僅差対策として、常時リングバッファに数秒分のフレームを保持し、検出後に前後フレームをまとめて解析する。

画像はROMや恒久ストレージには残さない。フルフレーム、切り抜き画像、僅差判定用スクショはすべて一時保存領域のみで扱い、ソフト終了時に削除する。

本番前提は Raspberry Pi 5 とし、横撮りで十分な視認性が確保できる設置位置を使う。

## 撮影姿勢の方針

今回は横から、またはゴールしてくる車両を右斜め前/左斜め前/右斜め後ろ/左斜め後ろから撮る前提で進める。ただし、順位判定だけを見ると真上からの方が有利で、横・斜め撮りは車両同士の重なりと遮蔽に弱い。

横から撮る利点:
- ゴールした瞬間の車両画像を人が見て確認しやすい。
- 切り抜き画像がそのまま確認画面に使いやすい。
- 車高や車体シルエットの差が見えやすい。

横から撮る弱点:
- 僅差で並んだときに前車が後車を隠しやすい。
- カメラに対して奥行き方向の誤差が出やすい。
- 土煙、草、背景遮蔽物の影響を受けやすい。

上から撮る利点:
- ゴールライン横断時刻を幾何的に取りやすい。
- 車両同士の重なりが減る。
- ゴールラインの仮想線を安定して置きやすい。

上から撮る弱点:
- 車両の見た目が似ていると人の目で区別しづらい。
- 切り抜き画像の視認性が落ちる。

対応するカメラ位置:
- 真横
- 車両の右前から見る
- 車両の左前から見る
- 車両の右後ろから見る
- 車両の左後ろから見る

設計判断:
- 本案件では「順位」と「通過時の画像確認」の両方が必要なので、初版は横・斜め撮りを採用する。
- 斜め撮りでは画面上のゴールラインが画面下側を横切る斜め線として写ることが多いため、スタート/ゴールラインは2点指定の任意線として扱う。
- ここでいう右前/左前/右後ろ/左後ろは、画面上の傾きではなく、ゴールしてくる車両から見たカメラ位置を指す。
- 横・斜め撮りで遮蔽が強く出る場合に備え、将来拡張として斜め上または2カメラ化を残す。

## 想定ハードウェア

- 開発環境: MacBook Air M1
- 本番: Raspberry Pi 5
- カメラ: 可能ならRaspberry Pi Global Shutter Cameraを優先。高速移動ではローリングシャッター歪みとブレが順位判定を悪化させる。
- 照明: 屋外/植物園跡地では影、葉、日照変化が大きいので、ゴールライン周辺だけでも補助照明を置く。
- 画角: 真横、車両の右前、左前、右後ろ、左後ろから選べるようにする。背景が単純になる位置を優先する。
- 背景: ゴールラインの背後に草木や人通りが入らない位置を優先する。

参考: Raspberry Pi Global Shutter Cameraは高速動体やマシンビジョン向けで、ローリングシャッター由来の歪みを避けやすい。短露光にも向く。

Raspberry Pi 5は Cortex-A76 2.4GHz CPU、LPDDR4X-4267 SDRAM を備え、Pi 4 より YOLO 実行向きである。

## 検出方式の候補

### A. OpenCV動体検出 + ゴールライン横断判定

ゴールライン周辺だけをROIとして切り出し、フレーム差分、背景差分、輪郭抽出、重心追跡で車両の通過を検出する。

利点:
- 軽いのでRaspberry Piでも高FPSを狙える。
- YOLOより検出遅延が小さい。
- 順位確定用の時刻を取りやすい。

弱点:
- 風で動く草、影、照明変化、水たまり反射などに弱い。
- 車両が重なった場合、1台として見えることがある。

対策:
- ROIをゴールライン周辺だけに限定する。
- 進行方向に沿った仮想ラインを2本置き、正しい方向に横断したものだけ採用する。
- 面積、速度、移動方向、連続フレーム数でノイズを除外する。

### B. YOLO常時検出

各フレームにYOLOをかけ、車両bboxの中心がゴールラインを横断したら記録する。

利点:
- 背景変化に比較的強い。
- bboxが取れるので切り抜き画像を作りやすい。

弱点:
- Raspberry PiではFPSが不足しやすい。
- ラジコンが近距離で高速通過すると、ブレや小さな写りで未検出が起きる。
- 推論完了順を順位に使うと僅差で順番が壊れる。

対策:
- 入力解像度を下げる。
- ゴールROIだけをYOLOに入れる。
- NCNN、TFLite、OpenVINOなど軽量形式を検証する。
- YOLOは順位確定ではなく、通過候補の確認と切り抜き補正に使う。

参考: UltralyticsのRaspberry Pi向け資料では、Raspberry PiではNCNN形式がARM向けに高速とされている。Raspberry Pi 5のベンチではYOLO26nのNCNNがONNXやTFLiteより速い結果になっているが、前処理/後処理時間は含まれないため実機検証が必要。

### C. ハイブリッド方式

OpenCVの動体検出で「何かがラインを通過した瞬間」を高速に検出し、その前後フレームを一時保存する。保存済みフレームを別スレッド/別プロセスでYOLOにかけ、車両らしさ、bbox、切り抜き画像を後から付与する。

利点:
- 順位は高速処理で確定できる。
- YOLOが遅れても順位が崩れない。
- 後処理で画像やID情報を充実させられる。

弱点:
- 実装は少し複雑になる。
- 動体検出の誤検出を後処理で棄却する仕組みが必要。

このプロジェクトではCを採用方式とする。

## YOLO運用方針

YOLOは順位の主判定器ではなく、次の用途で使う。

- OpenCVで拾った通過イベントに対する車両確認
- bbox補正
- 切り抜き画像生成
- 僅差時に複数台が写っているかの補助判定

### モデル選択

ユーザーが精度プロファイルを選べるようにする。採用するYOLO系統は `YOLO26` に固定する。

- `yolo26n`: Pi 5 上での新しい軽量候補。
- `yolo26s`: バランス候補。
- `yolo26m`: 高精度候補。ただしリアルタイム本番では重く、録画動画の再解析向け。

### 推奨プロファイル

- `Fast`: `yolo26n`
- `Balanced`: `yolo26s`
- `Accurate`: `yolo26m`

設計上の考え方:
- リアルタイム本番では `yolo26n` をデフォルトにする。
- `yolo26s` は環境に余裕がある場合のバランス設定として使う。
- `yolo26m` は高精度設定として搭載するが、Pi 5 では録画動画のオフライン再解析用が主用途になる。
- Pi 5 では YOLO26 系の NCNN が有望なので、YOLO11 系は初版には搭載しない。
- モデル切替は UI 設定から行い、変更時に再ロードする。

### 形式の方針

- Raspberry Pi 5 本番は NCNN を第一候補にする。
- MacBook Air M1 の検証では PyTorch または ONNX を許容する。
- 同じモデル名でも実体は `pt` と `ncnn` を切り替えられるようにする。

### 実装時の搭載モデル

- 本番標準: `yolo26n_ncnn`
- 本番バランス: `yolo26s_ncnn`
- 高精度解析: `yolo26m_pt` または `yolo26m_onnx`

### D. マーカー方式

車両の上面にArUco/AprilTag/色タグ/番号札を貼り、検出時にIDとして読む。

利点:
- 「どの車両か」をかなり確実に扱える。
- 学習データを作らなくても始められる。

弱点:
- 泥、ブレ、角度、遠さで読めないことがある。
- 見た目の自由度が落ちる。

今回は車両に物理マーカーを貼れない前提なので、この方式は採用しない。

### E. センサー併用

赤外線フォトゲート、レーザー、磁気、RFIDなどをゴールに置く。

利点:
- 僅差の順位確定がカメラ単体より強くなる。
- 暗さやブレに左右されにくい。

弱点:
- 車両ごとのIDや写真は別途カメラ処理が必要。
- 設置物がコース環境に影響する。

カメラ単体で厳しい場合の保険として検討する。

## ID管理方針

### フェーズ1: 通過イベントID

検出順に `E0001`, `E0002` のようなイベントIDを付ける。表示は順位、イベントID、通過時刻、切り抜き画像。

これは「その瞬間に通過した車両」を管理するIDであり、同じ車両を周回ごとに同一IDとして追跡するものではない。

### フェーズ2: 短時間トラックID

ゴールROI内で数フレームだけ重心やbboxを追跡し、`T001` のような一時トラックIDを付ける。ライン手前からライン後まで追跡できれば、通過時刻の補間精度も上がる。

### フェーズ3: 車両ID

車両ごとの固定IDが必要なら、見た目特徴ベースの再識別を検討する。ただし高速・泥・ブレ・横撮り遮蔽があるため、精度は限定的と考える。初版では固定車両IDは必須要件にしない。

## 順位確定ロジック

順位は次の順で確定する。

1. カメラ取得スレッドが全フレームに `frame_id` と単調増加時刻 `captured_at_ns` を付与する。
2. フレームをリングバッファに保存する。目安は通過前2秒、通過後1秒。
3. OpenCV検出でゴールライン付近の動体候補を作る。
4. 候補の重心またはbbox端がゴールラインを横断したフレームを見つける。
5. 横断前後の2フレームから線形補間し、推定通過時刻 `crossed_at_ns` を計算する。
6. `crossed_at_ns` 昇順で順位を確定する。
7. YOLOや切り抜き生成は後処理として実行し、順位そのものは変更しない。
8. OpenCVの通過候補画像に2台以上の車両候補が含まれる場合は `close_finish=true` として僅差検知を発火する。

僅差対策:
- 60fpsならフレーム間隔は約16.7ms、120fpsなら約8.3ms。これ以下の差はカメラ単体では確定が難しい。
- 僅差の場合は `確定`, `僅差`, `要確認` の状態を表示する。
- 同一フレーム内に複数候補がある場合は、ゴールラインに到達した位置を前後フレームで補間する。
- 本当に接戦を扱うなら、ゴールラインROIだけを高FPS・低解像度で撮るか、フォトゲートを併用する。

精度方針:
- 必要精度は未確定だが、高精度優先で設計する。
- そのため、短露光、高FPS、ROI限定、時刻補間、僅差検知、手動レビューを標準機能に含める。
- システムの絶対精度はカメラFPSとシャッター条件に支配されるので、実測ベンチマークで詰める。

### 僅差検知機能

次のどちらかを満たしたら僅差検知とする。

- OpenCVの通過トリガー時点の候補画像に2台以上の車両候補が見える。
- 上位2イベントの `crossed_at_ns` 差が閾値以内になる。

初期閾値:
- `close_finish_threshold_ms = 50`

理由:
- 60fps環境でも約3フレーム分に相当し、僅差候補を十分広く拾える。
- 120fps環境では約6フレーム分となり、手動確認対象を絞りすぎない。
- 最初は取りこぼし防止を優先し、誤って多めに `close_finish` へ落ちる方を許容する。

僅差検知時の動作:
- OpenCVトリガー時のスクショを一時保存する。
- 必要なら前後数フレームも一時保存する。
- そのスクショをUIに表示する。
- 対象イベントの `review_status` を `close_finish` にする。
- ソフト終了時、またはレースリセット時に削除する。

## ゴールラインの扱い

物理的なゴールラインは用意した方がよい。ソフト側でも仮想ゴールラインを設定するが、映像上で境界が曖昧だと検出が不安定になるため。

推奨:
- 路面に白線、テープ、マットなどで物理ラインを作る。
- UI上でその物理ラインに対応する仮想ラインを始点/終点の2点指定で設定する。
- 判定は仮想ラインで行う。
- 斜め撮りでは、画面内の白線やテープに沿って、画面下側を横切る斜めの仮想ラインを置く。

物理ラインがない場合:
- UI上の仮想ラインだけでも動く。
- ただし土の模様や影で境界が揺れるため、誤判定が増えやすい。

## レースモード

レースモードは2種類を持つ。

### 単純ゴール順モード

- スタート後、ゴールラインを最初に通過した順で順位を確定する。
- 指定台数がゴールした時点、または手動終了でレース終了にする。

### 周回モード

- レース開始前に周回数を設定する。
- 各車両のゴールライン通過をラップ通過として扱う。
- 最終周回のゴール通過順で最終順位を確定する。
- 同一車両の継続追跡は完全保証できないため、初版ではゴールROI近傍の短時間トラックと人のレビューを前提にする。
- 周回モードは単純ゴール順モードより難度が高いので、MVP内では対応するが制約付き機能とする。

### 周回モードで手動確認へ落とす条件

次のいずれかを満たしたら、その通過イベントは自動確定せず `manual_review` に落とす。

- 同じ周回候補時間帯に2台以上が検出され、どのトラックがどの既存車両に対応するか一意に決められない。
- ある車両の予測ラップ順と、実際の通過候補順が矛盾する。
- 前回観測から一定時間以上トラックが消失し、再出現した車両を同一個体として結び直す信頼度が低い。
- `close_finish` が発火し、かつ YOLO/OpenCV のどちらかで単独車両として分離できない。
- 連続2回以上のラップで同一車両IDの追跡が不安定になった。
- 既に完走済みとして扱った車両が再度ゴール候補として現れた。

初期の信頼度ルール案:
- トラック継続信頼度が `0.6` 未満なら `manual_review`
- 2秒以上の見失い後の再識別は `manual_review`
- 周回モードで `close_finish` になったイベントは既定で `manual_review`

運用上の考え方:
- 単純ゴール順モードでは多少攻めて自動確定してよい。
- 周回モードでは順位より「周回数の取り違え」が致命的なので、保守的に手動確認へ落とす。

## スタート制御

スタート方法は `自動` と `手動` の2モードを持つ。

### 自動スタート

- スタートラインまたはスタートROIを別途設定する。
- OpenCVでスタート通過を検知したらレースを開始する。
- 自動スタートが成立した時点で手動開始ボタンは無効化する。

### 手動スタート

- UIの開始ボタンでレースを開始する。
- 手動開始したレースでは自動スタート検知は無効にする。

### 状態遷移

- レース待機中: `自動開始待ち` または `手動開始待ち`
- 手動開始押下後: `running`
- 自動開始検知後: `running`
- `running` 中は開始方法の切り替え不可
- レース終了またはリセット後に再選択可

## 画像保存と表示

検出イベントごとに次を保持する。

- `event_id`
- `rank`
- `track_id`
- `crossed_at_ns`
- `frame_id`
- `bbox`
- `crop_path`
- `full_frame_path`
- `confidence`
- `review_status`

保存ポリシー:
- フルフレーム、切り抜き、僅差スクショはすべて一時ディレクトリに保存する。
- 永続保存は行わない。
- UI表示が終わっても同一レース中は参照できるように保持する。
- ソフト終了時、異常終了復旧時の次回起動時、レースリセット時に削除する。

表示画面:
- 現在のレース名
- 順位一覧
- 通過時刻
- イベントID/車両ID
- 切り抜き画像
- 判定状態
- 僅差検知スクショ
- 手動修正ボタン

## 実装アーキテクチャ

処理はリアルタイム系と後処理系を分ける。

```text
Camera Capture
  -> Frame Ring Buffer
  -> Fast Motion Detector
  -> Crossing Event Queue
  -> Ranking Engine
  -> UI / API

Crossing Event Queue
  -> Background YOLO Verifier
  -> Crop Generator
  -> Temp Image Store
  -> Metadata Storage
  -> UI Update
```

リアルタイム系:
- カメラ取得
- フレーム時刻付与
- 動体検出
- 横断時刻算出
- 順位確定

後処理系:
- YOLO検証
- bbox補正
- 切り抜き生成
- 一時画像保存
- レビュー用データ作成

## MacBook Air M1での検証計画

1. 録画動画でオフライン検証する。
2. ゴールラインROIを手動設定できるようにする。
3. OpenCV差分検出で通過イベントを出す。
4. イベント前後のフレームを保存し、順位が動画上の順番と一致するか確認する。
5. YOLOをROIだけにかけ、検出漏れと遅延を計測する。
6. 複数台・僅差・ブレ・影・草の揺れを含むテスト動画を作る。
7. Raspberry Pi実機でFPS、CPU温度、検出遅延を測る。
8. YOLOモデルごとに通過確認精度と処理待ち時間を比較する。

録画動画検証モードの要件:
- 動画ファイルを入力にできる。
- 再生速度、コマ送り、巻き戻しができる。
- 仮想ゴールラインを動画上で設定できる。
- 仮想スタートラインまたはスタートROIを設定できる。
- 通過イベント時刻、順位、僅差検知結果をあとから見直せる。
- 一時保存されたスクショと切り抜きをその場で確認できる。
- YOLOモデルを切り替えて再解析できる。

最初に作るもの:
- カメラまたは動画ファイル入力
- ゴールラインROI設定
- 仮想ゴールライン設定
- 仮想スタートラインまたはスタートROI設定
- OpenCV動体検出
